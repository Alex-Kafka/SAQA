# User Story

**User Story (Пользовательская история)** — это краткое, простое описание функции, рассказанное с точки зрения пользователя, который хочет получить от нее выгоду. Это не детальная спецификация, а **"обещание побеседовать"** для уточнения деталей.

### Структура (Шаблон Конбера)
**Как `<Роль>`, я хочу `<Что делать>`, чтобы `<Достичь цели/Получить выгоду>`.**

*   **Пример:** "Как *пользователь*, я хочу *сбросить свой пароль*, чтобы *получить доступ к аккаунту, если я его забыл*."

### Критерии приемки (Acceptance Criteria, AC)
Это условия, которые должны быть выполнены, чтобы Story считалась завершенной и работающей корректно. Часто оформляются как список.
*   **Пример:**
    1.  Пользователь получает email со ссылкой для сброса.
    2.  Ссылка действительна в течение 24 часов.
    3.  После смены пароля система разлогинивает пользователя на всех устройствах.

### Как понять, что User Story составлена правильно? (Правила INVEST)
-   **I**ndependent (Независимая): Минимизирует зависимости от других Story.
-   **N**egotiable (Обсуждаемая): Это тема для диалога, а не фиксированный контракт.
-   **V**aluable (Ценная): Предоставляет ценность для пользователя или бизнеса.
-   **E**stimable (Оцениваемая): Команда может оценить ее сложность.
-   **S**mall (Небольшая): Должна быть достаточно малой для реализации за один спринт.
-   **T**estable (Тестируемая): Имеет четкие [[Критерии приемки|критерии приемки]].

### Преимущества и недостатки (против Use Case)
| Преимущества | Недостатки |
| :--- | :--- |
| Простота и понятность | Могут быть слишком высокоуровневыми |
| Фокус на ценности для пользователя | Нехватка деталей для сложной логики |
| Идеально для итеративной разработки (Agile/Scrum) | Риск недопонимания без постоянного общения |

### Когда использовать?
- В **Agile-проектах** (Scrum, Kanban).
- Для описания большей части функциональности продукта.
- Когда команда имеет постоянный доступ к заказчику/продуктовому владельцу для уточнений.
