# Use Case

**Use Case (Вариант использования)** — это детальное, пошаговое описание взаимодействия между **Актором** (ролью) и **Системой** для достижения конкретной **Цели**. В отличие от [[User Story]], это законченная спецификация.

### Структура Use Case
1.  **Название:** Глагол + существительное (например, "Сброс пароля").
2.  **Актор(ы):** Кто инициирует взаимодействие (например, Пользователь).
3.  **Цель:** Что актор хочет достичь.
4.  **Предусловия:** Условия, которые должны быть истинны до начала сценария.
5.  **Постусловия:** Состояние системы после успешного завершения сценария.
6.  **Основной поток (Успешный сценарий):** Последовательность шагов от начала до успешного завершения.
7.  **Альтернативные потоки:** Отклонения от основного потока (ошибки, исключительные ситуации).

### Пример: "Сброс пароля"
1.  **Актор:** Пользователь.
2.  **Цель:** Восстановить доступ к аккаунту.
3.  **Предусловие:** Пользователь не авторизован в системе.
4.  **Основной поток:**
    1.  Пользователь переходит на страницу входа.
    2.  Пользователь нажимает ссылку "Забыли пароль?".
    3.  Система отображает форму для ввода email.
    4.  Пользователь вводит email и нажимает "Сбросить пароль".
    5.  Система генерирует уникальную ссылку, отправляет ее на email и отображает сообщение о проверке почты.
    6.  Пользователь переходит по ссылке в письме.
    7.  Система отображает форму для ввода нового пароля.
    8.  Пользователь вводит новый пароль дважды и нажимает "Сохранить".
    9.  Система проверяет пароль, сохраняет его, разлогинивает пользователя на всех устройствах и перенаправляет на страницу входа с сообщением об успехе.
5.  **Альтернативный поток А: Email не зарегистрирован.**
    *   На шаге 5: Система отображает сообщение "Указанный email не найден".

### Преимущества и недостатки (против User Story)
| Преимущества | Недостатки |
| :--- | :--- |
| Детальность и полнота | Требует больше времени на написание |
-   Четкое описание всех сценариев | Может быть избыточным для простых функций |
-   Меньше места для недопонимания | Менее гибкий, сложнее вносить изменения |

### Когда использовать?
- Для описания **критически важной и сложной бизнес-логики**.
- В проектах с **жесткими требованиями** (регуляторика, безопасность).
- Когда команда **распределена** и нет постоянного доступа к экспертам.
- Для документирования **"как есть"** процессов.
